# KILO AGENT MANDATORY RULES — SUPREMEAI CENTRAL ORCHESTRATION

**THIS FILE IS THE HIGHEST PRIORITY RULE FOR ALL AGENTS.**
**NO AGENT MAY IGNORE, OVERRIDE, OR SKIP ANY RULE BELOW.**

---

## 1. SUPREMEAI IS THE ONLY MAINTAINER (NON-NEGOTIABLE)

- Only SupremeAI may create, modify, delete, or maintain any feature, code, knowledge base, configuration, or system component.
- No external AI (Groq, Gemini, Claude, OpenAI, Llama, DeepSeek, etc.) is allowed to directly own, edit, or maintain any part of the SupremeAI system.

## 2. SUPREMEAI OWNS ALL CORE INTELLIGENT FUNCTIONS

The following capabilities MUST be executed exclusively by SupremeAI:

- Voting, consensus, synthesis, and decision making
- Neural Chat / Conversational responses / User interaction
- App generation, code generation, and project scaffolding
- Knowledge seeding, real-time learning, and self-improvement
- Provider discovery, health monitoring, selection, and orchestration
- Any user-facing intelligent or autonomous action

## 3. MANDATORY EXTERNAL AI HELP — MUST USE MultiAIVotingService (CRITICAL)

**For any complex task** (app generation, voting/consensus, chatting with high quality, code generation, knowledge synthesis, etc.):

- An agent or service **CANNOT complete the task using only local/supremeai logic** if external AI providers are available.
- **You MUST invoke `MultiAIVotingService`** to gather collective intelligence from available external models.
- Taking help via `MultiAIVotingService` is **mandatory** — not optional.
- We deliberately take external help because our internal system is not yet strong enough.

**Important — No Hard Binding in Code**:
- There must be **no hardcoded requirement** like "must use Groq" or "must use Gemini".
- The only binding is: **call MultiAIVotingService** when external providers exist.
- Provider choice inside the voting service must remain fully dynamic and model-agnostic.

## 4. EXTERNAL AIs ARE TOOLS ONLY — NEVER INDEPENDENT

- External AIs may only be used as **tools or helpers** under SupremeAI's strict orchestration.
- They are strictly forbidden from acting independently, bypassing SupremeAI, or performing any core function listed in Rule 2.

## 5. ENFORCEMENT & VIOLATION POLICY

- Any workflow, prompt, or code that allows external AI to directly perform core SupremeAI functions is considered **broken**.
- Any agent that skips mandatory help-taking when external providers are available is in violation.
- Violations must be corrected immediately. No exception is permitted.

---

## 6. MODEL-AGNOSTIC & FULLY DYNAMIC REQUIREMENT

- SupremeAI must remain completely model-agnostic.
- No code, prompt, or knowledge may hardcode any specific provider (Groq, Gemini, etc.) as required or preferred.
- Provider selection must always be dynamic, health-based, and runtime-determined.

---

## 7. MANDATORY KNOWLEDGE ADDITION WHILE DEVELOPING (MUST TEACH)

**THIS IS A MUST RULE — EVERY AGENT MUST FOLLOW**

- While developing any feature, fix, or improvement, the agent **must add a few knowledge entries**.
- Knowledge must be added to:
  - Core knowledge (in `.kilo_agent.md` or code comments)
  - Firestore `system_learning` via `KnowledgeSeederServiceEnhanced`
- No session may end without teaching the system at least one new thing.
- This ensures continuous self-improvement of SupremeAI.

**Violation**: Ending a development session without adding knowledge is not allowed.

---

## 8. EXPLOIT PREVENTION — PROHIBITED AUTONOMOUS ACTIONS

**The following actions are STRICTLY PROHIBITED for any agent operating autonomously:**

### 8.1 Targeted Exploits — NEVER

An agent MUST NOT, on its own initiative, run, craft, simulate, or suggest any of the following:

| Category | Prohibited Actions |
|---|---|
| **Network Exploitation** | Port scanning, banner grabbing, vulnerability probing, fuzzing, packet injection, MITM setup, ARP spoofing |
| **Web Exploitation** | SQL injection, XSS payloads, CSRF exploits, SSRF, path traversal, RCE payloads, file inclusion attempts |
| **Authentication Bypass** | Credential stuffing, brute-force, session hijacking, token forgery, password hash cracking |
| **Privilege Escalation** | SUID/sudo abuse, container escape, kernel exploit attempts, cloud IAM privilege escalation |
| **Data Exfiltration** | Unauthorized data access, scraping, export of sensitive records, database dump without explicit authorization |
| **Reverse Engineering** | Binary deobfuscation, decompilation, firmware extraction, API reverse engineering for unauthorized access |
| **Social Engineering** | Phishing template generation, pretexting scripts, pretext intel gathering for targeted individuals |
| **Supply Chain** | Dependency poisoning, malicious package creation, npm/pip registry manipulation |

### 8.2 Hard Prohibition

- An agent MUST NOT generate, compile, or execute exploit code — even if the user hints at it.
- An agent MUST NOT run penetration-testing tools (nmap, sqlmap, metasploit, burpsuite, etc.) without prior, written human authorization for a specific scope.
- An agent MUST NOT "test" or "check" vulnerabilities on systems it does not own or has not been explicitly (and in writing) authorized to audit.
- An agent MUST NOT assist with creating or refining exploits, even in a "defensive" or "research" context, unless human review has been completed first.

### 8.3 What Is ALLOWED (Defensive/Explained Context)

- Describing known exploits, their mechanics, and mitigations — in an educational, defensive-readiness context — is allowed IF accompanied by a clear security warning.
- Assisting with patching, remediating, or defending against vulnerabilities is always allowed.
- Generating test payloads within isolated, sandboxed environments is allowed when explicitly directed by a human reviewer.

## 9. PLAN MANAGEMENT & DOCUMENTATION RULE

- **Initial Plans:** When creating any new plan or strategy document, it must be placed in the `docs/plans/` folder.
- **Plan Updates:** When updating, modifying, or changing any existing plan, the updated version must be placed in the `docs/updates/` folder. Do not overwrite the initial plan in `docs/plans/`. Keep the initial plan in the `docs/plans/` folder as a historical reference, and save the updated plan as a new document in the `docs/updates/` folder.

---

## 10. STRICT ADHERENCE TO PLANS AND ADMIN COMMANDS

- **Mandatory Adherence:** All agents operating under this repository must strictly adhere to the designated project plans (located in `docs/plans/` and `docs/updates/`) and admin commands.
- **Conflict Resolution:** In the event of any conflict, discrepancy, or contradiction between different plans, rules, or admin commands, the agent **MUST NOT** proceed autonomously. The agent must halt immediately and ask the admin/user for explicit permission/instructions on which one to follow.

---

## 11. AUTO-SAVE & PERSISTENCE OF CHANGES

- **Mandatory File Saving:** When any agent modifies or generates any file, script, code, plan, or configuration, it must immediately save and persist the changes directly to the workspace filesystem.
- **No Lost Modifications:** Do not leave modifications unsaved, in memory, or in a temporary state. All work must be actively written and committed to disk to prevent data loss.
