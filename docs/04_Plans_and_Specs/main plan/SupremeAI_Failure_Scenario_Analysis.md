# SupremeAI: System Reliability & Failure Scenario Analysis

> **Status:** 🟢 Updated for v5 Architecture

এই ডকুমেন্টে ব্যাখ্যা করা হয়েছে কীভাবে SupremeAI সিস্টেম বিভিন্ন যান্ত্রিক ত্রুটি (Failure) মোকাবিলা করে এবং কীভাবে এর ফলব্যাক মেকানিজম কাজ করে।

---

## ১. সিস্টেমের প্রধান স্তরসমূহ (Core Layers)

1.  **External APIs (Level 1):** Groq, Gemini, OpenAI (ফ্রি টিয়ার)।
2.  **GCP Cloud Run / HF (Level 2):** আমাদের নিজস্ব হোস্ট করা ৫টি মডেল (Scale-to-Zero)।
3.  **Knowledge Base (Level 3):** আগে থেকে সেভ করা সমাধান বা নলেজ।
4.  **Backend System:** Spring Boot ব্যাকএন্ড যা পুরোটি পরিচালনা করে।

---

## ২. ধাপে ধাপে ফলব্যাক লজিক (Step-by-Step Fallback Logic)

সিস্টেমটি `AIFallbackOrchestrator` ব্যবহার করে নিচের ধাপগুলো অনুসরণ করে:

### ধাপ ১: নলেজ বেস চেক (Knowledge Check)

রিকোয়েস্ট আসার সাথে সাথে সিস্টেম দেখে এই সমস্যার সমাধান আগে থেকেই আমাদের ডেটাবেসে আছে কিনা। থাকলে এআই কল না করেই উত্তর দিয়ে দেয়।

### ধাপ ২: এপিআই কী রোটেশন (API Key Rotation)

যদি এপিআই কী ইনভ্যালিড হয়, সিস্টেম নিজে থেকে আরেকটি কী ব্যবহার করার চেষ্টা করে।

### ধাপ ৩: সার্কিট ব্রেকার (Circuit Breaker)

যদি কোনো প্রোভাইডার (যেমন Groq) বারবার ফেইল করে, সিস্টেম সেটিকে ৩০ সেকেন্ডের জন্য "ব্ল্যাকলিস্ট" করে দেয় এবং সরাসরি পরের প্রোভাইডারে চলে যায়।

---

## ৩. সম্ভাব্য ব্যর্থতা এবং সিস্টেমের প্রতিক্রিয়া (Failure Scenarios)

### পরিস্থিতি ক: যদি সব External API কী (OpenAI, Gemini) ফেইল করে

- **সিস্টেমের প্রতিক্রিয়া:** `AIFallbackOrchestrator` সাথে সাথে `dynamicChain` থেকে গুগল ক্লাউড বা হাগিং ফেসের আমাদের নিজস্ব এন্ডপয়েন্টগুলো (Qwen, Llama 3.1) বেছে নেবে।
- **ফলাফল:** ইউজার কোনো বাধা ছাড়াই উত্তর পাবেন, শুধু প্রথম রেসপন্সে কয়েক সেকেন্ড দেরি হতে পারে (Scale-to-Zero এর কারণে)।

### পরিস্থিতি খ: যদি আমাদের GCP Cloud Run মডেলগুলো ফেইল করে

- **সিস্টেমের প্রতিক্রিয়া:** যদি আমাদের নিজস্ব সার্ভারে কোনো ত্রুটি থাকে, সিস্টেম সাথে সাথে সার্কিট ওপেন করে দেবে এবং পুনরায় এক্সটারনাল এপিআইগুলোতে (যেমন Groq বা Mistral) রিকোয়েস্ট পাঠাবে।
- **ফলাফল:** সিস্টেমটি সবসময় বিকল্প পথ খুঁজে নেবে।

### পরিস্থিতি গ: যদি External API এবং GCP মডেল—উভয়ই ফেইল করে

- **সিস্টেমের প্রতিক্রিয়া:** এটি একটি "Critical Error" হিসেবে চিহ্নিত হবে। সিস্টেম তখন `RuntimeException` থ্রো করবে এবং ইউজারকে জানাবে যে সার্ভিসটি সাময়িকভাবে ডাউন। একই সাথে এডমিনকে অ্যালার্ট পাঠাবে।
- **ফলাফল:** সিস্টেমটি ভুল বা ভাঙা উত্তর (Toxic Code) জেনারেট করার চেয়ে নিরাপদভাবে বন্ধ হবে।

### পরিস্থিতি ঘ: যদি আমাদের ব্যাকএন্ড সিস্টেম (Spring Boot) ফেইল করে

- **সিস্টেমের প্রতিক্রিয়া:** গুগল ক্লাউড রান স্বয়ংক্রিয়ভাবে কন্টেইনারটি রিস্টার্ট করার চেষ্টা করবে (Self-healing)।
- **ফলাফল:** কয়েক সেকেন্ডের মধ্যে সিস্টেমটি পুনরায় সচল হয়ে যাবে।

---

## ৪. নিরাপত্তা স্তর: Code Immunity System

প্রতিটি উত্তর জেনারেট হওয়ার পর `immunitySystem.isCodeInfected()` মেকানিজম চেক করে যে কোডটি সঠিক কিনা। যদি কোডটি ভাঙা বা ক্ষতিকর হয়, সিস্টেম সেটি রিজেক্ট করে এবং অন্য কোনো মডেল দিয়ে পুনরায় চেষ্টা করে।

**উপসংহার:** SupremeAI এমনভাবে ডিজাইন করা হয়েছে যাতে এটি একটি "Self-Correction" লুপের মধ্যে থাকে। একটি বা দুটি সার্ভিস বন্ধ হয়ে গেলেও এটি বিকল্প পথে কাজ চালিয়ে যেতে সক্ষম।
